# Story 2.4: 入库错误处理与异常路径

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 用户,
I want 入库失败时看到清晰的错误信息,
so that 我能理解什么出了问题并决定下一步操作。

## Acceptance Criteria

1. **Given** 用户提供的文件路径不存在  
   **When** 执行 `kb ingest ./nonexistent.md`  
   **Then** 输出 `IngestionError`，包含失败步骤（`fetch`）、文件路径和“文件不存在”原因  
   **And** 退出码为 `1`
2. **Given** 用户提供的文件不是 Markdown 格式  
   **When** 执行 `kb ingest ./image.png`  
   **Then** 输出清晰错误提示，说明该文件类型不被支持，列出支持的格式  
   **And** 退出码为 `2`
3. **Given** pipeline 中 chunk 切分步骤失败  
   **When** 内容无法正确切分  
   **Then** 抛出 `IngestionError`，携带 `step='chunk'`、`source` 和具体原因  
   **And** pipeline 立即中止，不执行后续步骤  
   **And** 不产生部分写入的数据（事务回滚，NFR5）
4. **Given** 入库过程中用户按 `Ctrl+C` 中断  
   **When** 进程收到终止信号  
   **Then** 当前事务回滚，数据库保持一致状态（NFR5）  
   **And** 不产生残留的不完整记录
5. **Given** 用户尝试入库已存在的相同来源文件  
   **When** 系统检测到来源路径已存在于知识库中  
   **Then** 提示用户该内容已入库，显示已有条目信息  
   **And** 询问是否覆盖更新或跳过

### Review Findings

- [x] [Review][Decision] `SIGINT` 在同步 SQLite 事务执行期间无法被当前实现观测，现有实现只能保证“事务开始前取消不写入”，不能严格满足 AC4 的“当前事务回滚”语义。`better-sqlite3` 的 `provider.transaction(...)` 是同步阻塞执行，若信号在事务进行中到达，JS 回调会在事务提交后才有机会运行；经评审决定，本故事按“事务开始前取消不写入”的可接受语义关闭该项。
- [x] [Review][Patch] 重复来源分支会吞掉已发生的取消信号，错误返回成功/跳过文案 [`src/cli/commands/ingest.ts:125`]
- [x] [Review][Patch] 并发重复入库在非 TTY 下会退化为误导性的“请选择覆盖或跳过”错误 [`src/core/ingestion/pipeline.ts:166`]

## Dev Notes

- 这是 Epic 2 的第四个 story，目标是在已经可工作的本地 Markdown 入库链路上补齐**失败时的可理解性与数据一致性**，而不是重写 pipeline 或提前做 Epic 3 的通用输出体系。
- 当前仓库已经具备 `KbError` / `IngestionError`、`ProgressEvent`、TTY 进度渲染、标签备注写入和事务化存储；`2.4` 应在这条既有主干上增量扩展错误分类、重复入库处理和中断路径。
- 本 story 只覆盖 **本地 Markdown ingest** 的异常路径。网页抓取失败、HTTP 状态码、超时和正文提取失败属于 Epic 4，不要在这里顺手实现。

### Technical Requirements

- 保持现有技术栈与运行时约束：Node.js `^20.19.0 || >=22.12.0`、TypeScript strict、ESM 源码、`commander@14.0.3`、`better-sqlite3@12.9.0`、`@inquirer/prompts@8.4.2`、`vitest@4.1.4`。
- `@inquirer/prompts` 已在仓库中存在，并且 `src/cli/commands/init.ts` 已形成可注入 prompt adapter 模式；重复入库交互优先复用这一模式，不要额外引入新的交互库。
- 当前 `MarkdownAdapter` 仅支持 `.md`、`.markdown`、`.mdx`；“不支持的文件类型”需要输出稳定、可测试的支持格式列表，而不是只回显“未找到可处理该来源的入库 adapter”。
- 当前 `handleCliError()` 对所有 `KbError` 一律设为退出码 `1`；要满足 AC2，需扩展错误模型或映射规则，但不要在 core/pipeline 中直接设置 `process.exitCode`。
- 文件不存在错误不能只依赖 Node 原生 `ENOENT` 文案；需要在业务层明确包装出“文件不存在”的稳定语义，同时保留底层 `cause` 以便调试。
- 重复入库检测必须复用已有 `(source_type, source_path)` 唯一约束，并结合仓储查询输出已有条目摘要；不要把重复判断逻辑散落在 CLI、core、storage 多处重复实现。
- 当前 schema 已有 `chunks.knowledge_item_id -> knowledge_items.id ON DELETE CASCADE`、`item_tags.knowledge_item_id -> knowledge_items.id ON DELETE CASCADE` 与 `chunks_fts` 触发器；若实现“覆盖更新”，应利用这些现有约束，而不是手工维护多处删除顺序。

### Architecture Compliance

- 严格遵守依赖方向：`cli/ -> core/ -> storage/`。交互提示、退出码和信号处理位于 CLI 边界；业务编排位于 core；SQL 与数据查询位于 storage。
- 继续以 `KbError` 体系作为错误单一事实来源，不新增“字符串错误协议”或全局 logger；CLI 只负责将结构化错误格式化为用户可读输出。
- 继续以 `ProgressEvent` 作为 core -> CLI 的唯一过程可观测通道；错误步骤展示可以增强，但 core 层不得直接 `console.log()` 或 `process.stdout.write()`。
- 所有数据库写操作继续通过 Repository + `provider.transaction(...)` 完成；不要在 `core/` 或 `cli/` 直接写 SQL。
- 不要把本 story 演化成通用 formatter 重构、日志基础设施改造、全局事件总线或“版本化更新框架”。

### Current Implementation Context

- `src/cli/commands/ingest.ts` 当前已具备 `runIngestCommand()` 的依赖注入结构、TTY 判定、进度渲染接入和最终摘要输出，但没有重复入库交互，也没有针对输入错误的专门退出码分流。
- `src/core/ingestion/pipeline.ts` 已覆盖 `resolve-adapter -> fetch -> parse -> chunk -> store -> index` 六步，并在 `fetch` / `parse` / `chunk` / `store` 失败时发出 `error` 事件；其中 `fetch` 失败目前统一包装为“读取来源内容失败”，尚未明确区分“文件不存在”。
- `src/errors/index.ts` 当前 `KbErrorOptions` 仅包含 `step`、`source`、`cause`；若要支持退出码或交互决策，应优先考虑最小扩展，避免把 CLI 关注点过度渗透到 core。
- `src/cli/index.ts` 当前对 `CommanderError` 做单独退出码处理，但对所有 `KbError` 一律为 `1`；这是实现 AC2 的核心修改点。
- `src/adapters/markdown-adapter.ts` 已集中维护 Markdown 扩展名集合；支持格式列表应从这里或其等价单一来源派生，避免多处常量漂移。

### Implementation Guardrails

- 不要把“不支持的文件类型”继续伪装成通用 `resolve-adapter` 失败后就结束；需要让用户明确知道“你给的是 `.png`，当前只支持 `.md/.markdown/.mdx`”。
- 不要让非 TTY 场景进入交互等待；脚本、CI、Agent 调用必须是确定性行为。
- 不要在核心层直接 `process.exit()`、监听终端流或拼接最终人类可读文案；这些都属于 CLI 边界。
- 不要为了支持重复入库覆盖而顺手实现完整的“历史版本管理”“增量更新命令”或网页来源覆盖逻辑；本 story 只需要满足本地文件来源的最小闭环。
- 不要破坏当前成功路径的摘要格式、TTY 进度渲染与标签备注语义；`2.4` 是异常路径增强，不是重新定义 `kb ingest` 的成功输出。
- 不要在 repository 外部直接依赖 SQLite 唯一约束报错文本做用户提示；约束错误文本会随驱动/平台变化，必须转换成仓库内可控的业务语义。
