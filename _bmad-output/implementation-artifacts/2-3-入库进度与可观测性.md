# Story 2.3: 入库进度与可观测性

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 用户,
I want 在入库过程中看到实时处理进度,
so that 我知道系统正在工作并了解处理到了哪一步。

## Acceptance Criteria

1. **Given** 用户在 TTY 终端执行 `kb ingest ./doc.md`  
   **When** pipeline 各步骤执行  
   **Then** CLI 通过 `onProgress` 回调实时渲染各步骤状态（读取文件 → 内容清洗 → 切分 chunks → 存储入库 → 索引更新）  
   **And** 每个步骤显示开始、进行中、完成或失败状态
2. **Given** 入库完成  
   **When** 最后一个步骤执行结束  
   **Then** 输出结果摘要：标题、来源路径、内容字数、切分块数、已添加标签
3. **Given** 核心层函数  
   **When** 不注入 `onProgress` 回调  
   **Then** 静默执行，不产生任何终端输出（适配脚本/Agent 调用场景）

## Dev Notes

- 这是 Epic 2 的第三个 story，目标是在已经可工作的本地 Markdown 入库链路上补齐**运行时可见性**，而不是重做 pipeline 或推进搜索输出格式化。
- 当前代码里 `src/core/ingestion/pipeline.ts` 已经发出 `resolve-adapter -> fetch -> parse -> chunk -> store -> index` 的 `start/complete` 事件，但 `src/cli/commands/ingest.ts` 还没有真正消费这些事件；因此 `2.3` 的核心工作是把已有事件链接到 CLI 展示层，并只做这一层的必要增强。
- PRD 把“CLI 实时输出处理进度和结果摘要”定义在核心入库旅程里，因此本 story 完成后，`kb ingest` 应该第一次具备“看得见正在工作”的体验。

### Technical Requirements

- 技术栈保持当前仓库现状：Node.js 20+、TypeScript strict、ESM 源码、`commander@14.x`、`better-sqlite3@12.9.0`、`vitest@4.x`、`tsx@4.x`、`tsup@8.5.x`。
- 当前仓库尚未引入 CLI 渲染依赖；优先使用 Node 内置 TTY/ANSI 能力完成最小可靠实现。若确实需要第三方渲染库，需先证明它显著简化实现且不把 Epic 3 的格式化策略提前做掉。
- 外部资料校验结果：`chalk@5.6.2` 为当前稳定版本、ESM-only，兼容 Node 20；`ora@9.3.0` 需要 Node >=20，也兼容当前仓库运行时。即便如此，本 story 仍应默认选择**无新增依赖或最少新增依赖**的实现路径。
- `ingestSource()` / `ingestSourceWithProvider()` 的 `onProgress` 是 core 到 CLI 的唯一观测通道；不要引入第二套隐式日志、全局 emitter 或在 storage 层直写终端。
- 最终摘要输出继续沿用现有 `IngestResult`：`title`、`sourcePath`、`wordCount`、`chunkCount`、`tags`、`note`；不要在本 story 中把结果对象改造成 Epic 3 所需的通用 formatter 输入模型。
- 进度文案可以面向人类优化，但步骤语义必须稳定、可测试，避免把测试建立在易变的装饰字符或颜色码上。

### Architecture Compliance

- 严格遵守依赖方向：`cli/ -> core/ -> storage/`。进度渲染只能位于 CLI 层；`core/`、`storage/`、`adapters/` 不得依赖终端渲染实现。
- `ProgressEvent` 仍是结构化事件，不是格式化后的字符串。字符串拼装、颜色、光标控制、行清理都应留在 CLI 侧。
- 不要在本 story 中落地 `src/cli/formatters/index.ts`、`json-formatter.ts`、`plain-formatter.ts` 的完整策略体系；这些属于 Epic 3 的“输出格式化策略”。
- 可以为 ingest 单独新增轻量展示模块，但其职责必须聚焦“消费 `ProgressEvent` 并渲染终端状态”，不要趁机抽象整个 CLI 输出框架。
- 保持“不注入 `onProgress` 时静默执行”的架构约束，确保未来 MCP、脚本或测试场景仍能复用 core 层。
- 错误处理的单一事实来源仍是 `KbError` 体系与 `handleCliError()`；进度渲染只能补充上下文，不得替代最终错误输出。

### Current Implementation Context

- `src/core/ingestion/pipeline.ts` 已经在成功路径发出如下事件：`resolve-adapter:start/complete`、`fetch:start/complete`、`parse:start/complete`、`chunk:start/complete`、`store:start/complete`、`index:start/complete`。
- `src/core/types.ts` 中的 `ProgressEvent` 已支持 `step`、`status`、`detail`、`metadata`，状态枚举已包含 `start | progress | complete | error`，但当前 pipeline 还没有真正发送 `progress` 或 `error` 事件。
- `src/cli/commands/ingest.ts` 当前直接调用 `ingestSource()` 后以 `process.stdout.write()` 输出摘要，没有任何中间态显示，也没有 TTY/非 TTY 分支。
- `src/cli/commands/init.ts` 已有 `process.stdin.isTTY && process.stdout.isTTY` 的交互式判定模式；`2.3` 应复用这一判断方式，而不是新造一套环境检测规则。
- 当前仓库还没有 `src/cli/formatters/` 目录；如果在本 story 新增该目录，应只承载与 ingest 进度直接相关的内容，并避免让目录结构先于真实需求过度扩张。
- `tests/integration/ingestion.test.ts` 目前验证的是“成功入库并输出最小摘要”与“标签备注摘要”；引入进度显示后，这些断言需要根据 TTY/非 TTY 场景重新组织，避免测试因为附加过程输出而变得不稳定。

- 若需要测试 TTY 行为，优先通过可注入 writer / TTY 标志或轻量适配层做隔离，而不是在测试里硬依赖真实终端环境。

### Testing Requirements

- 单元测试至少覆盖：
  - `ProgressEvent` 到展示文案/状态的映射
  - 同一步骤多状态更新时的输出顺序和去噪策略
  - `error` 事件出现时的渲染行为
  - 非 TTY 模式下不注册或不输出进度
- pipeline 级测试至少覆盖：
  - 成功路径仍保持 `resolve -> fetch -> parse -> chunk -> store -> index` 的顺序
  - 关键失败路径会在抛错前发送对应步骤的 `error` 事件
  - 不传 `onProgress` 时依旧静默成功，不依赖终端存在
- CLI 集成测试至少覆盖：
  - TTY 场景：看到过程进度，并在结束后看到最终摘要
  - 非 TTY 场景：只输出最终摘要，不夹带过程日志
  - `--tag` / `--note` 摘要仍正确显示，避免 `2.2` 回归
- 测试断言应优先匹配语义文本，不要把 ANSI 颜色码、spinner 帧、光标控制序列写死成脆弱快照。

### Previous Story Intelligence

- `2.1` 已经把 `onProgress` 作为核心层预留接口加入函数签名，并通过测试确认成功路径事件顺序；`2.3` 不需要重新设计 pipeline，只需要把这条能力真正用起来。
- `2.1` / `2.2` 已经建立了清晰分层：CLI 负责参数和摘要，core 负责编排，storage 负责 SQL；进度展示必须继续遵守这条边界，不能因为“只是显示一下”就把逻辑回灌进 core。
- `2.2` 已经让成功摘要包含标签和备注，因此 `2.3` 的完成定义不仅是“看见进度”，还包括**不能破坏现有成功摘要内容**。
- 当前 `pipeline.test.ts` 已验证成功事件序列，但尚未覆盖 `error` 事件；这是本 story 最直接的测试增量入口。

### Implementation Guardrails

- 不要把 `onProgress` 的字符串文案直接硬编码在 core 层之外多处重复定义；应在一个明确位置维护“步骤 -> 展示名称”映射，避免后续 `2.4` / `4.x` 改名时多处漂移。
- 不要在非 TTY 场景下也持续打印进度日志，否则会污染脚本输出，并破坏 PRD/架构里“脚本/Agent 调用场景保持静默”的要求。
- 不要为了做进度显示就提前引入完整表格、颜色主题、JSON formatter、全局 logger、EventEmitter 总线或通用日志基础设施。
- 不要把错误详情在 `onProgress` 和 `formatKbError()` 里重复输出两遍；进度负责“哪一步失败了”，错误处理负责“为什么失败”。
- 不要修改 `IngestResult` 的核心语义，也不要改变 `kb ingest` 现有参数协议；`2.3` 是可观测性增强，不是命令接口重构。
