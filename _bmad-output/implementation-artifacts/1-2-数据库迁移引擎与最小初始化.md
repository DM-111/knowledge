# Story 1.2: 数据库迁移引擎与最小初始化

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 用户,
I want 工具在首次需要持久化能力时自动创建数据库并完成最小初始化,
so that 我无需手动设置数据库，同时后续功能只在真正需要时引入对应 schema。

## Acceptance Criteria

1. **Given** 用户首次执行任何需要数据库的命令  
   **When** 命令启动并解析出数据库路径  
   **Then** 系统在配置路径下自动创建 SQLite 数据库文件  
   **And** 通过 `PRAGMA user_version` 检测并顺序执行所有未应用的迁移脚本
2. **Given** 最小初始化迁移（如 `001-bootstrap.sql`）执行  
   **When** 迁移完成  
   **Then** 数据库具备迁移引擎运行和当前已启用能力所需的最小结构  
   **And** 不预先创建 `chunks`、`tags`、`item_tags`、`chunks_fts` 等后续能力才需要的数据库对象
3. **Given** 后续故事首次引入 chunk、tag 或 FTS 能力  
   **When** 对应能力被实现  
   **Then** 该故事同时引入对应的编号迁移脚本  
   **And** 新 schema 通过渐进式迁移加入，而不是回填到初始迁移中
4. **Given** 迁移脚本执行过程中发生错误  
   **When** 某个迁移失败  
   **Then** 该迁移的事务被回滚，数据库保持上一个已知好的状态  
   **And** 抛出 `StorageError`，包含失败的迁移版本号和原因
5. **Given** `DatabaseProvider` 接口  
   **When** 在测试环境中使用  
   **Then** 可以注入 `:memory:` 数据库实例，不依赖文件系统

## Dev Notes

- 这是 Epic 1 的第二个实施 story，目标是把“存储初始化能力”从零变成可复用的基础设施，但仍然严格停留在最小初始化层，不抢后续 story 的业务实现。
- 当前 `1.1` 已经交付 CLI 骨架、统一错误模型、Node `>=20` 基线和 CJS 构建产物；`1.2` 应直接复用这些基础，不要重新定义入口、错误约定或测试风格。

### Previous Story Intelligence

- `1.1` 已建立 ESM 源码 + CJS 构建产物模式；源码 import 使用 `.js` 后缀指向编译后模块，新增存储层文件也应保持同样写法。
- CLI 顶层已统一处理 `CommanderError`、`KbError` 和未知异常；本 story 的所有迁移/初始化失败都应进入 `StorageError` 分支，而不是裸 `throw new Error(...)`。
- 当前集成测试风格使用 `vitest` + `execa('node', ['bin/kb.js', ...])`，中文测试描述，`reject: false` 后断言 `exitCode` 与输出内容；如果需要 CLI smoke test，沿用该模式。
- `src/storage/index.ts` 目前仍是空导出，占位正好说明本 story 应在该层建立公开 API，但不要越界到真正的 Repository 或业务表实现。

### Technical Requirements

- 存储实现必须基于 `better-sqlite3`，以同步 API 保持事务边界清晰，符合架构文档既定选型。
- 迁移版本跟踪必须使用 SQLite 原生 `PRAGMA user_version`；不要自建额外 migration history 表，也不要引入第三方迁移库。
- 迁移文件命名遵循 `NNN-description.sql`，从 `001-bootstrap.sql` 开始，按数字升序执行。
- 每个迁移必须在独立事务中执行，且“执行迁移 SQL + 更新 `user_version`”应视为同一原子单元处理，避免版本号与 schema 脱节。
- `DatabaseProvider` 需要为未来 Repository 模式留出口，但本 story 只实现迁移和初始化所需的最小接口；不要在这里提前发明完整数据访问层。
- 文件型数据库路径由上层传入；本 story 可以假设“调用方已经解析出 dbPath”，但不负责实现完整配置解析或交互式补参流程。
- 若需要创建数据库父目录，应在存储层内部完成，确保首次落盘路径可用；失败时统一包装为 `StorageError` 并带上 `source=dbPath`。

### Architecture Compliance

- 严格遵守依赖方向：`cli/ -> core/ -> storage/`。`cli/` 不能直接 new `better-sqlite3`，也不能直接扫描 `migrations/`。
- 本 story 只交付“迁移引擎 + 最小 bootstrap”；不得提前实现以下内容：
  - Story `1.3` 的 `kb init` 交互式配置、配置文件读写、四层配置合并
  - Epic `2` 的 `knowledge_items` / `chunks` / FTS 入库 schema 与 pipeline
  - Epic `3` 的检索 schema、查询构建、结果格式化
  - Epic `5` 的标签 schema、shell completion
- `001-bootstrap.sql` 的职责是让数据库具备“可继续演进”的基础，而不是让后续所有能力一次性可用。
- 如果需要在 CLI 中增加一个极薄的触发路径来验证自动建库，也只能调用 core 暴露的初始化入口；该路径依旧可以保持业务占位，不应顺手实现真实 `list/search/init` 行为。

### Library / Framework Requirements

- `better-sqlite3`：当前稳定版本可按 `12.9.0` 基线理解，Node.js 要求为 `>=20`；实现时仍以包管理器实际解析结果为准。
- `tsup`：继续沿用当前构建器，不在本 story 擅自替换；但必须把 `better-sqlite3` 作为 external 处理，避免 native addon 打包/运行异常。
- SQLite：`PRAGMA user_version` 是官方提供的应用保留版本字段，适合作为编号迁移的单一事实来源；不要再维护第二套版本状态。
- 测试依然使用 `vitest`；如需要验证文件型数据库，可使用临时目录；如验证 provider/migrator 逻辑，优先使用 `:memory:` 降低测试成本。

### Testing Requirements

- 至少覆盖以下场景：
  - `:memory:` provider 可成功初始化并跑完迁移
  - 文件型数据库在目标路径不存在时可被创建
  - 首次启动执行 `001-bootstrap.sql` 后，`PRAGMA user_version` 为 `1`
  - 二次启动不会重复执行已应用迁移
  - 失败迁移会回滚，数据库停留在上一个已知好的版本
  - `001-bootstrap.sql` 不包含 `chunks`、`tags`、`item_tags`、`chunks_fts`
- 测试重点是“迁移机制正确”和“最小初始化边界正确”，不是数据库业务表 CRUD。
- 若编写 CLI 级 smoke test，断言目标应是建库与迁移行为，而不是借机实现 `kb init` 或 `kb list` 的真实用户体验。

### Latest Technical Information

- `better-sqlite3` 在 2026 年仍为活跃维护方案，`12.9.0` 为当前稳定线之一，且 Node.js 运行时基线已提升到 `>=20`；这与当前仓库 `package.json` 的 Node 版本约束一致。
- `PRAGMA user_version` 的官方用途就是为应用保存一个整数版本号，因此非常适合按序编号迁移；Story 1.2 应把它作为唯一 schema 版本来源。
- 对于 `tsup` + `better-sqlite3` 组合，行业通行做法仍是将 `better-sqlite3` 设为 external，让运行时从依赖安装位置加载原生模块，而不是尝试打进 bundle。
