# Story 2.2: 入库时标签与备注支持

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 用户,
I want 在入库时通过 `--tag` 和 `--note` 选填标签和备注,
so that 我可以在入库时就对知识进行初步分类和标注。

## Acceptance Criteria

1. **Given** 用户执行 `kb ingest ./doc.md --tag "typescript,学习笔记" --note "关于泛型的总结"`  
   **When** 入库流程完成  
   **Then** 知识条目关联指定的标签（通过 `item_tags` 多对多关系）  
   **And** 备注信息存储在 `KnowledgeItem` 记录中  
   **And** 入库结果摘要中展示已添加的标签和备注
2. **Given** 标签不存在于数据库中  
   **When** 入库时指定了新标签  
   **Then** 系统自动创建新标签记录并建立关联
3. **Given** 用户未提供 `--tag` 和 `--note`  
   **When** 入库流程完成  
   **Then** 知识条目正常入库，标签和备注为空  
   **And** 不阻塞入库流程

## Dev Notes

- 这是 Epic 2 的第二个 story，目标是在已经跑通的本地 Markdown 入库链路上补上**入库时元数据增强**，而不是重做 ingestion pipeline。
- 当前 `2.1` 已经建立了 `resolve -> fetch -> parse -> chunk -> store -> index` 的主干、`knowledge_items`/`chunks`/`chunks_fts` schema、CLI 最小摘要输出，以及 `provider.transaction(...)` 的事务边界；`2.2` 应该在这些既有能力上增量扩展。
- 本 story 只覆盖 FR7 在 **ingest 路径**上的落地。已存在条目的标签编辑属于 Epic 5，按标签检索/浏览属于 Epic 3，实时进度与异常矩阵属于 `2.3` / `2.4`。

### Technical Requirements

- 技术栈与风格保持当前实现：ESM 源码、import 使用 `.js` 后缀、`better-sqlite3` 持久化、`vitest` 测试、`tsup` 构建。
- CLI 入口继续复用 `ensureConfigForCommand()`，不要在 `ingest` 命令里重复实现配置加载或直接打开数据库连接。
- `IngestSourceOptions` 可以新增 `tags?: string[]`、`note?: string` 一类字段，但不要把标签/备注需求塞进 `Markdown adapter` 的职责边界里。
- 标签需要支持“指定不存在标签时自动创建再关联”；推荐在仓储层提供 `ensureTags()` / `linkTagsToItem()` 之类的显式方法，而不是在多个位置散落查找/插入逻辑。
- 备注应作为 `KnowledgeItem` 的可选字段持久化；字段命名需与数据库 `snake_case` / 代码 `camelCase` 的约定一致，在 Repository 层完成映射。
- `--tag` 的解析规则要明确、可测试：逗号分隔、trim、忽略空项；是否去重应在 story 实现中一并定义，避免写入重复关联。

### Architecture Compliance

- 严格遵守依赖方向：`cli/ -> core/ -> storage/`。`src/cli/commands/ingest.ts` 不能直接 import Repository 或 `better-sqlite3`。
- 所有 SQL 只能位于 `src/storage/`（migrations / repositories）；`src/core/ingestion/pipeline.ts` 只负责编排，不直接写 SQL。
- 标签、备注、chunk、FTS 同属本次入库的持久化结果，必须保持在同一事务边界内，满足 NFR5 的一致性要求。
- 迁移必须使用新编号 SQL 文件渐进式引入 schema，不允许回填 `001-bootstrap.sql` 或重写既有迁移。
- `onProgress` 回调接口保持现状即可；本 story 不负责设计 TTY 富文本进度渲染，也不需要补齐 `2.4` 的完整错误分支矩阵。

### Current Implementation Context

- `src/core/ingestion/pipeline.ts` 当前在事务内创建 `KnowledgeItem` 与 `Chunks`，并通过现有返回值向 CLI 暴露标题、来源、字数、chunk 数；这是接入标签/备注的主扩展点。
- `src/storage/repositories/knowledge-item-repository.ts` 当前 `INSERT` 字段仅包含 `title`、`source_type`、`source_path`、`content`、`word_count`、`created_at`，尚未包含备注列。
- `src/cli/commands/ingest.ts` 当前只输出四行摘要：标题、来源、字数、切分块数；新增标签/备注时需要在这里扩展结果展示。
- `tests/integration/storage-ready.test.ts` 当前显式断言数据库中**不存在** `tags` / `item_tags`；2.2 落地后必须同步修正，否则会制造假失败。
- `src/cli/commands/tag.ts` 目前仍是占位入口；不要因为本 story 引入 `tags` 表就提前把 `kb tag` 做成完整功能。

### Previous Story Intelligence

- `2.1` 已明确把 `--tag` / `--note`、实时进度渲染、Web ingestion、搜索格式化排除在范围外；`2.2` 的职责是把其中“入库时标签与备注”这一个增量闭环补上。
- `2.1` 已经通过 `002-ingestion-schema.sql` 引入 `knowledge_items`、`chunks`、`chunks_fts`；因此 `2.2` 新增的 `tags` / `item_tags` / 备注字段应继续遵守“首次需要时再引入”的渐进式迁移原则。
- `2.1` 产出的代码模式已经稳定：CLI 做参数与摘要、core 做编排、storage 做 SQL；`2.2` 不应打破这条分层边界。

### Implementation Guardrails

- 不要在 `Markdown adapter` 中解析或持久化标签/备注；adapter 仍然只负责把源内容标准化成 `RawContent`。
- 不要为了图省事把标签当作逗号拼接字符串塞进 `knowledge_items`；应使用独立 `tags` 与 `item_tags` 多对多关系。
- 不要把“首次标签创建”“标签去重”“关联去重”逻辑写在 CLI 层；CLI 只负责解析输入，不负责决定持久化细节。
- 不要顺手实现 `kb tag`、`kb search --tag`、`kb list --tag`、`--json` 摘要输出或交互式补标签流程，这些都不属于本 story 的完成定义。
